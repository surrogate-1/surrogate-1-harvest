# surrogate-1 / design

### Diagnosis
* The project lacks a robust implementation for handling Hugging Face API rate limits, which can block dataset training.
* There is inadequate reuse of existing Lightning Studio instances, leading to wasted quota and potential downtime.
* The data ingestion pipeline does not effectively utilize the HF CDN bypass, resulting in unnecessary API calls and potential rate limit issues.
* The project does not have a mechanism to prevent Lightning idle stop from killing training processes.
* The code does not handle the HF commit cap, which can block ingestion and training.

### Proposed change
The proposed change is to implement a robust HF API rate limit handler, reuse existing Lightning Studio instances, utilize the HF CDN bypass, prevent Lightning idle stop from killing training processes, and handle the HF commit cap. The scope of this change includes modifying the `train.py` script and the `ingestion.py` script to incorporate these improvements.

### Implementation
To implement this change, the following steps can be taken:
1. Modify the `train.py` script to use the HF CDN bypass for downloading dataset files, instead of making API calls. This can be achieved by using the `https://huggingface.co/datasets/{repo}/resolve/main/{path}` URL to download files directly.
2. Implement a mechanism to reuse existing Lightning Studio instances. This can be done by listing the existing studios and reusing the ones that are already running.
3. Add a rate limit handler to the `ingestion.py` script to handle the HF API rate limit. This can be achieved by implementing a retry mechanism with a delay of 360 seconds after receiving a 429 response.
4. Modify the `train.py` script to check the status of the Lightning Studio instance before each `.run()` call and restart the instance if it has stopped.
5. Implement a mechanism to handle the HF commit cap by spreading writes across multiple sibling repositories.
